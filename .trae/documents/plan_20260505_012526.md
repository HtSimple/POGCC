## 具备自我规划能力的外部知识检索 Agent 实现方案

### 核心思路

将当前线性 RAG 流程升级为 **LangGraph 条件路由 Agent**，让 LLM 自主决定：
1. 是否需要检索外部知识？
2. 从哪里检索（本地知识库 vs 网络搜索）？
3. 检索到的知识是否足够？不够则继续检索
4. 何时可以生成最终回答？

### 新的 Agent 工作流

```
用户查询
  ↓
[规划节点] ← LLM 分析查询，决定下一步行动
  ↓
  ├─→ "直接回答" → [生成回答] → END
  ├─→ "检索本地知识库" → [本地检索] → [评估节点]
  ├─→ "网络搜索" → [网络搜索] → [评估节点]
  └─→ "两者都需要" → [本地检索] → [网络搜索] → [评估节点]
                                        ↓
                                  [评估节点] ← LLM 判断知识是否充分
                                        ↓
                                   ├─→ 充分 → [生成回答] → END
                                   └─→ 不充分 → 回到[规划节点]（换策略重试）
```

### 实现步骤

#### 1. 创建搜索工具服务 `app/services/search_service.py`
- 集成 Tavily Search API（免费额度足够开发使用）
- 封装为统一接口 `search(query, max_results=5)`

#### 2. 创建工具定义 `app/core/rag_agent/tools.py`
- 定义两个 LangGraph 工具：`local_search`（本地知识库检索）和 `web_search`（网络搜索）
- 工具描述清晰，让 LLM 能正确选择

#### 3. 重构 Agent `app/core/rag_agent/agent.py`
- 新增 `planner_node`：LLM 分析查询意图，选择工具
- 新增 `evaluate_node`：LLM 评估检索结果是否充分
- 新增 `web_search_node`：执行网络搜索
- 改造 `retrieve_documents_node`：执行本地检索
- 使用条件边（conditional edges）实现路由
- 设置最大重试次数防止死循环

#### 4. 更新配置 `config.json.example`
- 添加 `tavily_api_key` 字段

#### 5. 更新依赖 `requirements.txt`
- 添加 `tavily-python` 包
