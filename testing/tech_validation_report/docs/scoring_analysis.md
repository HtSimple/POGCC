# S1-S8 场景评分分析

## 评分维度定义

1. **主题相关性**（1-5）：内容是否紧扣给定主题，不跑题、不泛化
2. **大纲遵循度**（1-5）：生成内容是否严格遵守大纲结构，章节顺序、页数、要点是否对应
3. **内容完整性**（1-5）：是否覆盖关键要点，信息密度与深度是否充足
4. **结构清晰度**（1-5）：章节逻辑、页面过渡、单页层次是否清晰流畅
5. **知识补充利用程度**（1-5）：是否有效利用检索知识/上传文档，证据引用是否恰当
6. **事实一致性风险**（1-5，5=风险最低）：是否存在矛盾陈述或无证据支撑的断言
7. **PPT展示可用性**（1-5）：少量修改是否即可直接用于PPT展示

## 生成失败详情

8个场景共64个slide，其中62个成功（96.9%），2个失败。

### 失败1：S4 slide-003（重点行业：电力、钢铁水泥、交通物流）

**失败原因**：Pydantic Schema 字段长度校验失败。`PageContentProtocol` 协议要求 `slides[].evidencePack` 最多包含 5 条证据，但 LLM 为该页生成了 6 条证据引用。

**根因分析**：LLM 在生成该页时自行将 `researchPolicy.maxSourcesPerSlide` 设为 6（超出协议默认上限 5），导致 `evidencePack` 实际包含 6 条引用。内容本身质量很高——涵盖了本地政策文档、政府规划、学术论文和行业报告等多种来源，均有具体的数据和 quote。失败纯粹由后端 Pydantic schema 的 `max_length=5` 限制触发，与内容生成质量无关。

**验证报错原文**：
```
structured content validation failed: 1 validation error for PageContentProtocol
slides.0.evidencePack
  List should have at most 5 items after validation, not 6 [type=too_long]
```

### 失败2：S7 slide-004（效率瓶颈与转型需求，对应 outline 中的 slide-004 增效目标页）

**失败原因**：Pydantic Schema 字段长度校验失败。`PageContentProtocol` 协议要求 `researchPolicy.sourcePriority` 最多包含 5 个来源类型，但 LLM 为该页生成了 6 个来源类型（`local_document, official_sites, government_reports, academic_sources, authoritative_media, industry_reports`）。

**根因分析**：S7 为无文档场景，但 LLM 在 `sourcePriority` 中错误地包含了 `local_document`（该场景未上传任何文档），同时列出了 6 种来源类型超过协议上限。该页的 core message 和 displayBullets 实际上已生成（"流程冗余、协同不畅与数字化水平不足，正严重制约企业运营效率..."），但结构化校验拒绝后 `page_content` 被置为 null。

**验证报错原文**：
```
structured content validation failed: 1 validation error for PageContentProtocol
researchPolicy.sourcePriority
  List should have at most 5 items after validation, not 6 [type=too_long]
```

### 两例失败的共性

| 共性问题 | 说明 |
|----------|------|
| 根因 | LLM 在自主决定 `maxSourcesPerSlide` / `sourcePriority` 规模时可能超出 Pydantic schema 约束 |
| 影响 | 均为字段长度校验失败，非内容生成失败；实际 JSON 内容已完整生成在 `content` 字段中 |
| 修复方向 | 方案1：放宽 schema 字段上限（如 evidencePack → max 8, sourcePriority → max 8）；方案2：在 prompt 中明确约束上限值；方案3：在解析层做截断容错 |

## 改进建议

1. **Schema容错**：放宽 `evidencePack` 和 `sourcePriority` 的 max_length（建议8），或在解析层做截断容错而非直接拒绝
2. **数据虚构防控**：对于无文档支撑的结论导向型场景，应在prompt中明确要求LLM标注"以下数据为示例/估算"或使用占位符（如[需补充实际数据]）
