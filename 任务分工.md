# POGCC 后续任务分工说明

本文档基于当前项目状态和前期任务概述，对后续 11 项工作进行详细说明。每项任务包含目标、主要工作、涉及模块、交付物、验收标准与风险依赖，便于小组成员分工、排期和验收。

## 一、整体目标

POGCC 的目标是完成一个可部署、可演示、链路完整的“PPT 大纲智能生成与内容补全系统”。系统应支持：

- 用户创建 PPT 生成任务。
- 上传或登记参考资料，并通过本地 RAG 检索使用。
- 生成结构化 PPT 大纲，并允许用户修改。
- 对每页进行知识补充、正文生成、演讲备注生成和事实一致性检查。
- 输出最终 Markdown 文本。
- 保存历史记录，支持回溯每个步骤的具体情况。
- 记录模型调用、Token、成本和性能指标。
- 具备基础质量验证、测试和部署能力。

## 二、任务详情

## 1. RAG 代码合并

### 目标

将当前项目中分散的 RAG 相关代码整合到统一链路中，使“资料入库、文本切分、向量化、检索、返回来源”形成稳定闭环。

### 主要工作

- 将 KnowledgeAgent 的本地检索接入正式链路，形成“先本地 RAG，必要时外部搜索”的策略。
- 保证 `/api/rag/upload`、`/api/rag/query` 能稳定调用合并后的 RAG 服务。
- 删除或标记废弃的旧代码，避免后续维护混乱。

### 交付物

- 合并后的 RAG 服务代码。
- 明确的 RAG 调用接口。

### 验收标准

- 能上传/登记至少一种文档类型并成功入库。
- 查询时能返回相关内容摘要。
- 查询结果能包含来源信息。
- RAG 模块测试通过。
- 文档解析失败时要有明确错误信息。

## 2. JSON 规约合并

### 目标

统一 PPT 大纲、页面内容、知识检索结果、事实检查结果和最终 Markdown 组装所依赖的数据结构，避免前后端和 LLM 输出格式不一致。

### 交付物

- 更新后的 `app/schema/models.py`。
- JSON Schema 文档或注释。
- 前端 `types.ts` 类型同步更新。
- 大纲生成、内容生成、事实检查的结构化输出示例。

### 验收标准

- 后端所有生成接口返回结构化 JSON。
- Pydantic 校验失败时能返回明确错误。
- 前端无需猜测字段格式即可渲染。
- 测试覆盖合法 JSON、缺字段、字段类型错误等情况。

## 3. 数据库部分开发

### 目标

为部署和历史记录提供后端持久化能力，替代仅依赖浏览器 localStorage 的临时方案。

### 当前基础

当前前端历史记录主要保存在浏览器 localStorage，后端暂未提供任务、文档、页面内容和历史记录的正式数据库表。

### 主要工作

- 选择数据库方案：
  - 开发和课程演示阶段推荐 SQLite。
  - 部署阶段可迁移到 PostgreSQL 或 MySQL。
- 设计基础表。
- 增加数据库初始化和迁移脚本。
- 后端接口改为以数据库中的 `project_id` 贯穿全流程。
- 历史记录从后端查询，而不是只从前端 localStorage 读取。

### 交付物

- 数据库模型定义。
- 初始化脚本或迁移脚本。
- 项目、文档、页面、历史记录相关 CRUD 服务。
- 数据库配置文档。

### 验收标准

- 后端重启后任务历史仍可查询。
- 同一任务的资料、大纲、正文、备注和最终文本能完整回溯。
- 数据库文件或连接配置可在部署环境中配置。
- 基础 CRUD 测试通过。

## 4. 备注生成功能开发

### 目标

为每页 PPT 生成 Speaker Notes，使演讲者能获得比幻灯片正文更详细的讲解内容。

### 主要工作

- 新增后端接口，例如：
  - `POST /api/generator/notes`
- 新增请求结构：
  - `project_id`
  - `slide_id`
  - `slide_title`
  - `slide_content`
  - `knowledge_evidence`
  - `style_requirement`
- 新增响应结构：
  - `success`
  - `notes`
  - `message`
- 编写专门的备注生成提示词。
- 备注应满足：
  - 不直接复制正文。
  - 更适合口头表达。
  - 能补充背景、解释概念、串联上下文。
  - 不新增无来源事实。
- 将备注写入数据库中的对应 slide 记录。
- 前端将“生成备注”改为调用专用接口。

### 交付物

- 备注生成服务。
- 备注生成 API。
- 提示词模板。
- 前端按钮和状态更新。
- 测试用例。

### 验收标准

- 任意一页能根据正文和证据生成备注。
- 备注内容明显区别于幻灯片正文。
- 备注能保存并在历史记录中查看。
- 生成失败时前端有清晰反馈。

### 风险与依赖

- 如果正文质量较差，备注会被连带影响。
- 备注可能引入无来源扩展内容，需要与事实检查联动。

## 5. 后端链路整体逻辑修改

### 目标

将后端从“多个零散能力接口”升级为“围绕 project/task 的完整生成流程”，支持状态流转、持久化和质量控制。

### 主要工作

- 大纲修改后保存为正式版本，后续内容生成只能基于确认后的大纲。
- 页面级生成独立执行，允许失败重试。
- 本地 RAG、外部搜索、内容生成和事实检查串联为统一服务层。
- 错误处理统一返回格式。
- 所有关键步骤写入数据库和日志。

### 交付物

- 重构后的后端流程服务。
- 项目状态流转逻辑。
- 统一错误结构。
- 历史步骤记录。
- 对应接口文档。

### 验收标准

- 一个 `project_id` 可以贯穿完整生成流程。
- 后端能阻止不合法状态跳转，例如未生成大纲就生成最终文本。
- 每一步都能查询当前状态和结果。
- 异常失败后可重试，不影响已完成步骤。

## 6. 后端接口设计与对应简单前端开发

### 目标

设计稳定、清晰、前端易用的 API，并开发与之匹配的简单前端流程。

### 建议接口设计

推荐后续接口围绕 `/api/projects` 设计：

| 方法   | 路径                                                      | 说明                 |
| :----- | :-------------------------------------------------------- | :------------------- |
| `POST` | `/api/projects`                                           | 创建 PPT 任务        |
| `GET`  | `/api/projects`                                           | 查询历史任务列表     |
| `GET`  | `/api/projects/{project_id}`                              | 查询任务完整详情     |
| `POST` | `/api/projects/{project_id}/documents`                    | 添加资料并入库       |
| `POST` | `/api/projects/{project_id}/outline`                      | 生成大纲             |
| `PUT`  | `/api/projects/{project_id}/outline`                      | 保存用户修改后的大纲 |
| `POST` | `/api/projects/{project_id}/slides/{slide_id}/knowledge`  | 补充页面知识         |
| `POST` | `/api/projects/{project_id}/slides/{slide_id}/content`    | 生成页面正文         |
| `POST` | `/api/projects/{project_id}/slides/{slide_id}/notes`      | 生成页面备注         |
| `POST` | `/api/projects/{project_id}/slides/{slide_id}/fact-check` | 页面事实一致性检查   |
| `POST` | `/api/projects/{project_id}/markdown`                     | 生成最终 Markdown    |

### 前端主要工作

- 将当前 localStorage 历史记录改为调用后端历史接口。
- 每一步完成后保存到后端。
- 前端仍保持两个主流程：
  - 生成 PPT。
  - 查看历史记录。
- 根据后端状态控制下一步按钮。
- 历史记录详情展示每一步具体结果。

### 交付物

- 后端路由实现。
- 前端 API 客户端实现。
- 前端页面联调。

## 8. 提示词优化

### 目标

优化大纲生成、内容生成、备注生成、知识融合、事实检查和最终 Markdown 组装相关提示词，提高输出稳定性和质量。

### 当前基础

当前提示词集中在：

- `app/prompts/templates.py`

已有大纲、内容、RAG、搜索规划、评估和整理相关模板。

### 主要工作

- 按功能拆分提示词：
  - 大纲生成提示词。
  - 大纲修复提示词。
  - 页面知识查询提示词。
  - 页面正文生成提示词。
  - 演讲备注生成提示词。
  - 事实一致性检查提示词。
  - 最终 Markdown 组装提示词。
- 明确输出格式，尽量要求 JSON。
- 为每个提示词加入：
  - 输入字段说明。
  - 输出字段说明。
  - 禁止事项。
  - 示例输出。
- 加强约束：
  - 不编造来源。
  - 不输出 schema 之外字段。
  - 没有证据时要标记不确定。
  - 正文和备注区分表达方式。
- 建立提示词版本记录，方便回滚和对比。

### 交付物

- 优化后的提示词模板。
- 典型输入输出样例。
- 提示词版本说明。
- 对比测试结果。

## 9. 成本 / Token 管控代码及文档

### 目标

记录并控制每次 LLM 调用的 Token、耗时、模型、重试次数和估算成本，为性能优化和部署预算提供依据。

### 主要工作

- 在 LLM 服务层统一封装调用记录。
- 每次调用记录：
  - `project_id`
  - `step`
  - `provider`
  - `model`
  - `prompt_tokens`
  - `completion_tokens`
  - `total_tokens`
  - `estimated_cost`
  - `latency_ms`
  - `retry_count`
  - `success`
  - `error_message`
- 支持不同 provider 的价格配置。
- 如果 API 不返回 Token，则至少记录估算 Token。
- 增加任务级总成本统计。
- 前端可展示当前任务的调用次数、Token 和估算成本。
- 编写成本控制文档。

### 交付物

- LLM 调用日志代码。
- 成本估算配置。
- 数据库记录表。

## 10. 整体性能及效果测试

### 目标

对系统端到端流程进行性能和效果评估，确认系统能稳定完成演示和部署运行。

### 测试范围

- 文档入库耗时。
- RAG 查询耗时。
- 外部搜索耗时。
- 大纲生成耗时。
- 页面正文生成耗时。
- 备注生成耗时。
- 事实检查耗时。
- 最终 Markdown 组装耗时。
- 端到端完整任务耗时。

### 效果测试维度

- 大纲结构是否合理。
- 页面数量是否符合用户要求。
- 内容是否覆盖主题。
- 检索结果是否相关。
- 正文是否适合 PPT 展示。
- 备注是否适合演讲。
- 事实检查是否能发现问题。
- 最终 Markdown 是否完整、层级清楚。

### 主要工作

- 准备 3 到 5 个标准测试主题。
- 每个主题配套至少 1 份参考资料。
- 记录每一步耗时和成功率。
- 记录人工评分结果。
- 对比不同模型表现。
- 输出性能与效果测试报告。

### 交付物

- 性能测试脚本。
- 效果测试样例。
- 测试报告。
- 问题清单和优化建议。

### 验收标准

- 至少完成 3 条端到端测试流程。
- 每条流程都有耗时、Token、质量评价记录。
- 系统失败点和优化方向明确。
- 演示场景能稳定跑通。

### 5. 集成优先级要求

第 5、6 项依赖其他模块能够正常跑通。为避免整体进度阻塞，各模块负责人应先提交“简单但可使用”的版本合并到项目中，后续再分别优化。

具体要求：

- RAG 模块先保证能完成基本入库和查询，不必一开始就追求最优检索效果。
- JSON 规约先确定核心字段，后续可逐步扩展质量指标和成本字段。
- 数据库先支持任务、页面和历史记录的基本保存查询。
- 备注生成先提供可调用接口，后续再优化提示词和质量控制。
- 网络搜索先完成基础搜索、去重和摘要返回，后续再补权威性排序和复杂过滤。
- 成本 / Token 管控先记录基础调用日志，后续再完善价格估算和统计页面。
- 前端先保证能调用后端接口跑通主流程，视觉和交互细节可后续优化。

### 6. WBS 补充说明

WBS 中一些相对独立、边界清晰的功能由张昊天补充和完善，比如：“API 故障自动重试”等。

需要注意：“事实一致性检验”不归入 API 重试类工作，而应归入第 7 项质量验证，由叶贤伟负责主导。

## 四、统一验收清单

最终项目完成时，应满足以下条件：

- 前端只有“生成 PPT”和“查看历史记录”两个主流程。
- 生成流程按步骤线性推进，不能跳过关键步骤。
- 后端以 `project_id` 贯穿任务全流程。
- 大纲、页面内容、备注、事实检查结果均有结构化 JSON。
- 参考资料可入库并被 RAG 检索。
- 外部搜索具备过滤、去重和摘要融合能力。
- 每页可生成正文和演讲备注。
- 每页可进行事实一致性检查。
- 最终 Markdown 可生成、下载、保存。
- 历史记录可查看每一步具体情况。
- LLM 调用有 Token、耗时、成本记录。
- RAG、搜索、生成、事实检查有测试覆盖。
- 系统可部署到云服务器并完成演示。
